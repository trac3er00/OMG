VARIANT ASSIGNMENT STRATEGIES
=============================

1. Hash-based (deterministic):
   - Uses SHA256(experiment_id:subject_id) % num_variants
   - Same subject always gets same variant
   - Useful for consistent user experience

2. Random (default):
   - Uses random.choice(variants)
   - Each call may assign different variant
   - Useful for true A/B testing

IMPLEMENTATION NOTES
====================

1. Shared In-Memory DB Handling:
   - For :memory: DBs, maintain persistent connection in __init__
   - Closing last connection to shared :memory: DB destroys it
   - Solution: Keep _shared_memory_conn alive for lifetime of ExperimentTracker
   - Only close file-backed DB connections in finally blocks

2. Connection Pattern:
   - Connection-per-invocation for file-backed DBs (same as TelemetryCollector)
   - Persistent connection for in-memory DBs (test support)
   - All connections use check_same_thread=False for subprocess safety

3. Schema Initialization:
   - Called once per file-backed connection (idempotent via CREATE TABLE IF NOT EXISTS)
   - Called once in __init__ for shared in-memory connection
   - Schema version stored in schema_info table for future migrations

4. Error Handling:
   - ValueError for invalid variants (< 2) or assignment strategy
   - ValueError for missing experiment_id in assign_variant()
   - RuntimeError for feature flag disabled
