## Task 2: Pinned Trust Root + Ed25519/Minisign Signing Core

- Local trust is now pinned to `registry/trusted_signers.json`; verification resolves signer by `key_id` from that file and does not trust artifact-carried public key material.

## Task 5 (T5): Signed Approval Artifact Support For Advisory Outcomes

- `ApprovalArtifact` dataclass binds approval to a specific `artifact_digest` (SHA-256 hex) plus `action`, `scope`, `reason`, `signer_key_id`, `issued_at`, and `signature`.
- Signing reuses `sign_artifact()` from T2 with a custom `signing_payload` parameter — the payload includes a `type: omg-approval-artifact-v1` field to namespace the signing domain and prevent cross-type signature replay.
- The `signature` field is excluded from the signing payload (security-critical invariant) — `_approval_signing_payload()` builds the canonical dict without it.
- Verification resolves the signer from `registry/trusted_signers.json` (pinned trust root) — unknown key IDs and algorithm mismatches produce explicit rejection.
- `verify_approval_artifact()` accepts both `ApprovalArtifact` dataclass and plain `dict` inputs, normalizing internally.
- `load_approval_artifact_from_path()` returns a 3-key dict: `{valid, reason, approval}` — `approval` is `None` on failure.
- Tests reuse the same `_DEV_PRIVATE_KEY` / `_DEV_KEY_ID` constants as `test_warn_and_run.py` for consistency.
- Pre-existing LSP errors in `hooks/config-guard.py`, `tests/hooks/test_validator_v2.py` — not caused by T5, not touched.

## Task 6 (T6): Compliance Governor Hardening

- `evaluate_release_compliance()` no longer has release-evidence pass-through; missing `release_evidence` or missing `artifact` now blocks at artifact gate.
- `_artifact_gate_decision()` now enforces strict action handling: only `allow` passes directly; `ask` requires a valid signed approval artifact bound to `subject_sha256`; all unknown actions block.
- Approval support accepts either inline `approval_artifact` (verified via `verify_approval_artifact`) or `approval_artifact_path` (loaded via `load_approval_artifact_from_path`), and allowed responses carry audit fields plus approval metadata.
- Release-level allowed responses now retain artifact audit context (`artifact_alg`, `artifact_key_id`, `artifact_subject_sha256`, `artifact_verdict`) and optional approval fields, so downstream callers can audit final compliance outcomes.
- Control-plane `registry_verify` now maps `decision.action == "deny"` to HTTP 403 for transport-level blocking semantics.
